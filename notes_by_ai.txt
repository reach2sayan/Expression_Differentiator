● The eval_seeded approach doesn't scale — let me trace why.                
                                                                                                                                                                                 
  forward_mode_gradient runs static_for<N> and for each pass J, builds a seeded array s, then calls expr.template eval_seeded<symbols>(s). The array lookup vals[idx] inside     
  every Variable::eval_seeded is a runtime indirection — the compiler can't fold through it the same way it folds through a direct member read.                                  
                                                                                                                                                                                 
  For F1/F2 the expression tree is small enough that the compiler inlines through everything anyway (~0.2 ns). For F4 (4 variables, x*y*z*w makes a wide tree, mixed             
  exp/sin/polynomial), the combined instantiation size of static_for<4> × nested static_for<4> × the full F4 eval_seeded tree blows past the compiler's inlining budget — it
  starts emitting real function calls at ~10 ns each. 53 ns / 4 passes ≈ 13 ns per pass, exactly what you'd expect from a few non-inlined calls.                                 
                                                                                                                                                                               
  jacobian_forward_mode() avoids this by doing mutate-evaluate-restore: seed variables directly, call plain eval() (which reads a struct member — trivially inlineable), then    
  restore. The scalar gradient<DiffMode::Forward> should do the same. The trade-off is it becomes non-const (it mutates and restores the expression), same as the reverse-mode
  hessian already does.                                                                                                                                                          
                                                                                                                                                                               
  Want me to rewrite detail::forward_mode_gradient to use the mutate-evaluate strategy? It should bring F4 back in line with the other modes.

● The fix is in place and all tests pass. Here's the summary:                                                                                                                    
                                                                                                                                                                                 
  What was broken: forward_mode_gradient used eval_seeded with a runtime for loop. For large expression trees (F4, 4 variables), Clang hit its inlining budget and generated a   
  slow 53.6 ns path; GCC handled it fine but a static_for fix regressed Clang on F2.                                                                                             
                                                                                                                                                                                 
  Root cause of the whack-a-mole: GCC and Clang have opposing inlining preferences for this pattern:                                                                             
  - GCC needs __attribute__((always_inline)) on the inner per-pass lambdas to see through them and constant-fold compile-time inputs                                             
  - Clang constant-folds each pass independently when lambdas are a reasonable size; forcing them all inline via always_inline merges 4 copies of the expression tree into one   
  function that exceeds Clang's optimization threshold                                                                                                                        
                                                                                                                                                                                 
  The fix (include/gradient.hpp): one outer std::index_sequence lambda (no Boost mp_for_each), an init fold to set all seeds, then a fold of inner per-pass lambdas. A
  DIFF_PASS_INLINE macro applies always_inline on GCC only.                                                                                                                      
                                                            
  Results: both compilers now match autodiff's forward-mode speed — ~0.23 ns across F1, F2, and F4 on both GCC and Clang release builds, 166/166 tests still passing.
